The OSI L7: Application

The Application Layer involves the protocols used by programs to exchange data, regardless of the underlying network.

L2 is link-to-link, L3 is end-to-end, L4 is process-to-process. L7 is application-to-application.

It is the closest to the user and the one actually producing the data content.

TCP/IP vs ISO/OSI Application Layer

The TCP/IP Application Layer functionality is splitted into 3 layers in the ISO/OSI framework.

Application Layer protocols in the TCP/IP protocol suite includes HTTP, FTP, IMAP, POP, DNS, SMTP, DHCP, Telnet, TFTP, etc.

Presentation and Session Layers

Layer 6 - Presentation Layer

Its purpose is to represent data in a common way:

  • It encodes data in a compatible format, understandable by the destination.
  • It handles data compression and decompression.
  • It encrypts and decrypts for transmission/reception.

It's about file formats and standards (like JPG, PNG, MPEG).

Layer 5 - Session Layer

L5 protocols initiate (authenticating and authorizing, if necessary), keep active, terminate or restart dialogues (sessions) between source and destination applications.

Examples are: PAP, SCP, RPC, PPTP, L2TP, etc.

The Client-Server Model

In the client-server model:

  • the host requesting the information is a client.
  • the host responding to the request and providing the information is a server.

The client begins the exchange by requesting data from the server, which responds with stream(s) of data to the client.

Application layer protocols describe the format of these exchanges between clients and servers.

In addition to the actual data (L6) transfer, these exchanges (L7) may require user authentication and session management (L5).

Server software is often implemented as a background service that waits for requests by the clients.

Peer-to-Peer (P2P) Model

In the P2P network model devices are peers to one another: they carry both roles (client and server) simultaneously.

Each peer can assume the role of the client for a particular request, while acting as a server role for another one.

In the P2P network model, P2P Applications are essential to coordinate the roles of the peers without centralizing accounts, permissions, data.

They can also operate as an hybrid: the actual sharing of the resources (file, peripherals, etc) is P2P-based and decentralized, but the resource indexes are stored on 1 or more centralized servers. eMule and early BitTorrent are notable examples.

Common P2P Applications

P2P applications often consist of a background service and the User Interface.

HyperText Transfer Protocol (HTTP)

Browsers interpret the URL parts, recognize the type of data and organize retrieval with the server using the right protocol.

Web pages are identified by their URL (Uniform Resource Locator). When we type an URL into our browser, pages are requested and served by the HTTP Protocol.

HTTP is a request/response protocol. It operates exchanging messages and status codes. The 3 most important messages are GET, POST and PUT. Releavant status codes include 200 OK, 304 Not Modified, 404 Not Found.

HTTP is not a secure protocol and its exhanges are transmitted in plain text, easily interceptable. HTTPS combines HTTP with Secure Socket Layer (SSL) and Transport Secure Layer (TLS) to provide authentication and encryption.

How E-Mail Works

Email is a store-and-forward messaging system.

Email clients never communicates directly with one another, they communicate with servers to send and receive email.

Mail servers communicate between them to relay messages from domain to domain (or even in the same domain).

The protocol used for sending mail (both client-to-server and server-to-server) is the SMTP Protocol.

Clients' MUAs (Mail User Agent) do mail retrieval from servers using either POP or IMAP protocols.

All these mail protocols needs proper configuration of the server address in the MUA.

Simple Mail Transfer Protocol (SMTP)

SMTP server software runs TCP port 25 (587 for Secure SMTP).

The message format used by SMTP requires at least 2 headers (sender and recipient addresses) and a message body.

Other headers exist, but are optional.

When the SMTP server receives a message, it:

  • stores it on a local account, if the recipient is local.
  • forwards it to another mail server, using SMTP.
  • schedules mail for retransmission if the destination server is offline or not reachable. After a certain expiration time, the server replies to the sender with an undeliverable notification.

Post Office Protocol

Server listen for POP connection by clients on TCP Port 110.

Generally POP servers delete the messages after client retrieval, making it not ideal for centralized backups and account synchronization.

Internet Message Access Protocol

IMAP server software runs on TCP Port 143 (993 for IMAPS).

Messages are always kept on the server until manual deletion. Clients make a local copy of their inbox on the server.

It's not only message content that is synchronized with the server, but also folder hierarchy and other metadata.

Synchronization is 2-way, so a mailbox server through IMAP is seamlessly usable on different devices at once.

Domain Name System (DNS)

DNS was created to solve the problem of having to remember long numeric addresses. It matches domain names to network addresses and performs address resolution.

The DNS protocol defines:

  • a message format that is used for all kinds of DNS communications (client queries, server responses, error messages, DNS records transfer betweeen servers).
  • DNS records and their purposes.

The most popular DNS server software is the Berkeley Internet Name Domain (BIND) daemon.

How DNS works

  • Upon receiving a client's query, the server DNS daemon search in its own DNS records for a match to that name.
  • If there's none, it contact other DNS servers until it is able to resolve the name.
  • It then sends a DNS reply to the client containing the resolved address, but also put the acquired data into a cache for faster responses and less bandwidth consumption.

DNS Hierarchy

DNS uses a hierarchical system to build the name-to-address mapping database. It looks like a reverted tree.

The database is splitted into small, manageable DNS zones. If a server has records corresponding to a certain level in the hierarchy, it is said to be authoritative for those records.

The hierarchy is made by Root Servers, TLDs (Top-Level Domain), secondary and so on.

DNS Records

  • A - Single host address.
  • NS - Authoritative name server.
  • CNAME - It maps a single network address to multiple names that act as an alias. It called canonical name or Fully Qualified Domain Name.
  • MX - Used to list the mail servers associated and responsible for exchanging mail for that domain.

Dynamic Host Configuration Protocol

DHCP allows hosts to autoconfigure network parameters such IP address, subnet mask, default gateway and DNS servers.

Addresses are dynamically leased from and address pool. An address no longer used is returned to the pool for reallocation.

A DHCP server keeps track of leased addresses in the configured range, assuring their uniqueness.

DHCP Operations

  1. Client broadcasts a DHCP discover (DHCPDISCOVER) that will reach any available DHCP server.
  2. Server replies with a DHCP offer (DHCPOFFER), proposing a lease and providing all network parameters and lease duration.
  3. Client sends a DHCP request (DHCPREQUEST) precisely to the server and lease offer it is accepting.
  4. Server returns a DHCP acknowledgment, positive or negative (DHCPACK or DHCPNAK), finally registering the lease.

Client can also renew their lease before expiration, or request a previously assigned address through a DHCPREQUEST.

File Transfer Protocol (FTP)

FTP is a protocol that allows for pushing and pulling data to and from a server. FTP opens 2 connections:

  • One on port 21 is just for connection control, commands and replies between client and server.
  • A second one for the actual data transfer.

Server Message Block (SMB)

SMB (formerly CIFS) is just a client/server sharing protocol that can share more than just files, but also printers, serial ports and inter-process messages.

Unlike FTP, it is made for long-term connections that make the resources accessible just as if they were local.

Since Windows 2000, SMB services use DNS for name resolution, abandoning the previous non-TCP/IP solution (NETBIOS).

*nix systems have their implementation of the SMB protocol called SAMBA.
